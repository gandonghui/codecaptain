# CodeCaptain-core Server

CodeCaptain runs on top of an CodeCaptain-core server. By default it starts one for you, so you don't have to do anything. You only need this page if you want to point CodeCaptain at a server you already run, or manage the one it starts.

## How CodeCaptain finds a server

When CodeCaptain starts, it looks for a server in this order:

1. reuse a server it already started
2. connect to an external one if you told it to (see below)
3. auto-detect a server on the default port (`4096`)
4. otherwise, start and manage its own

If nothing is configured, step 4 happens automatically and you're up and running.

## Connect to a server you already run

Set these before starting CodeCaptain:

```bash
OPENCODE_HOST=http://localhost:4096 OPENCODE_SKIP_START=true codecaptain
```

- `OPENCODE_HOST` — the full address of your CodeCaptain-core server, including the port (a value like `http://localhost:4096`). It must not have a path at the end.
- `OPENCODE_SKIP_START=true` — tells CodeCaptain not to start its own server.

If you only need to change the port, set `OPENCODE_PORT` instead of `OPENCODE_HOST`.

If `OPENCODE_HOST` is missing its port or has a path, CodeCaptain ignores it and falls back to starting its own server. Watch the startup logs for a `[config]` warning if a connection you expected didn't happen.

## Manage the server from the CLI

```bash
codecaptain status
codecaptain logs
codecaptain restart
codecaptain stop
```

`codecaptain` on its own starts the server in the background. Add `--foreground` to keep it attached to your terminal.

## Start CodeCaptain at login

Use `startup enable` to install a native user service. CodeCaptain uses `launchd` on macOS, `systemd --user` on Linux, and Task Scheduler on Windows.

```bash
codecaptain startup enable
codecaptain startup status
codecaptain startup disable
```

To protect the UI, set the password when enabling the service:

```bash
CODECAPTAIN_UI_PASSWORD='secret' codecaptain startup enable
```

For a headless server that starts at login and is meant for desktop or mobile clients, include `--api-only` and a reachable host:

```bash
codecaptain startup enable --port 3000 --api-only --host 0.0.0.0 --ui-password secret
```

`startup enable` snapshots your current environment into the service so it behaves more like starting `codecaptain` from the same shell. This keeps provider tokens, `PATH`, SSH agent settings, and other CLI auth/config variables available. Use `--no-env-snapshot` if you want a minimal service environment.

The startup service remembers `--port`, `--host`, `--ui-password`, and `--api-only`. CLI restart and update restart reuse those saved settings.

To create a connection link for another CodeCaptain app, use:

```bash
codecaptain connect-url --port 3000 --server http://your-host:3000 --qr
```

Run `codecaptain connect-url --help` to see all link options, including `--name`, `--lan`, `--server`, `--api-only`, `--ui-password`, and `--qr`.

You can still manage tunnels independently for that running service:

```bash
codecaptain tunnel start --port 3000
codecaptain tunnel stop --port 3000
```

Stopping the tunnel does not restart the service or the app.

## "CodeCaptain-core is restarting"

While the server is starting or restarting, CodeCaptain shows an "CodeCaptain-core is restarting" state and pauses requests until it's ready. This is normal right after launch or a restart. If it never clears, see [CodeCaptain-core connection](/troubleshooting/opencode-connection/).
