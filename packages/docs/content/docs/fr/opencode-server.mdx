# Serveur CodeCaptain-core

CodeCaptain fonctionne par-dessus un serveur CodeCaptain-core. Par défaut, il en démarre un pour vous, donc vous n’avez rien à faire. Vous n’avez besoin de cette page que si vous voulez pointer CodeCaptain vers un serveur que vous exécutez déjà, ou gérer celui qu’il démarre.

## Comment CodeCaptain trouve un serveur

Au démarrage, CodeCaptain cherche un serveur dans cet ordre :

1. réutiliser un serveur qu’il a déjà démarré
2. se connecter à un serveur externe si vous le lui avez indiqué (voir ci-dessous)
3. détecter automatiquement un serveur sur le port par défaut (`4096`)
4. sinon, démarrer et gérer son propre serveur

Si rien n’est configuré, l’étape 4 se fait automatiquement et vous pouvez commencer.

## Se connecter à un serveur que vous exécutez déjà

Définissez ces variables avant de lancer CodeCaptain :

```bash
OPENCODE_HOST=http://localhost:4096 OPENCODE_SKIP_START=true codecaptain
```

- `OPENCODE_HOST` — l’adresse complète de votre serveur CodeCaptain-core, port inclus (par exemple `http://localhost:4096`). Elle ne doit pas se terminer par un chemin.
- `OPENCODE_SKIP_START=true` — indique à CodeCaptain de ne pas démarrer son propre serveur.

Si vous voulez seulement changer le port, définissez `OPENCODE_PORT` au lieu de `OPENCODE_HOST`.

Si `OPENCODE_HOST` n’a pas de port ou contient un chemin, CodeCaptain l’ignore et revient au démarrage de son propre serveur. Surveillez les logs de démarrage pour un avertissement `[config]` si une connexion attendue ne se fait pas.

## Gérer le serveur depuis la CLI

```bash
codecaptain status
codecaptain logs
codecaptain restart
codecaptain stop
```

`codecaptain` seul démarre le serveur en arrière-plan. Ajoutez `--foreground` pour le garder attaché à votre terminal.

## Lancer CodeCaptain à la connexion

Utilisez `startup enable` pour installer un service utilisateur natif. CodeCaptain utilise `launchd` sur macOS, `systemd --user` sur Linux et le Planificateur de tâches sur Windows.

```bash
codecaptain startup enable
codecaptain startup status
codecaptain startup disable
```

Pour protéger l’UI, définissez le mot de passe lors de l’activation du service :

```bash
CODECAPTAIN_UI_PASSWORD='secret' codecaptain startup enable
```

Pour un serveur headless qui démarre à la connexion et doit servir des clients desktop ou mobile, ajoutez `--api-only` et un host joignable :

```bash
codecaptain startup enable --port 3000 --api-only --host 0.0.0.0 --ui-password secret
```

`startup enable` capture votre environnement actuel dans le service, pour qu’il se comporte davantage comme un lancement de `codecaptain` depuis le même shell. Les tokens de fournisseurs, `PATH`, les réglages d’agent SSH et les autres variables d’auth/config CLI restent ainsi disponibles. Utilisez `--no-env-snapshot` si vous voulez un environnement de service minimal.

Le service de démarrage mémorise `--port`, `--host`, `--ui-password` et `--api-only`. Les redémarrages CLI et les redémarrages après mise à jour réutilisent ces réglages enregistrés.

Pour créer un lien de connexion pour une autre application CodeCaptain, utilisez :

```bash
codecaptain connect-url --port 3000 --server http://your-host:3000 --qr
```

Lancez `codecaptain connect-url --help` pour voir toutes les options de lien, dont `--name`, `--lan`, `--server`, `--api-only`, `--ui-password` et `--qr`.

Vous pouvez toujours gérer les tunnels séparément pour ce service en cours d’exécution :

```bash
codecaptain tunnel start --port 3000
codecaptain tunnel stop --port 3000
```

Arrêter le tunnel ne redémarre ni le service ni l’application.

## « CodeCaptain-core redémarre »

Pendant que le serveur démarre ou redémarre, CodeCaptain affiche l’état « CodeCaptain-core redémarre » et met les requêtes en pause jusqu’à ce qu’il soit prêt. C’est normal juste après un lancement ou un redémarrage. Si l’état ne disparaît jamais, consultez [Connexion à CodeCaptain-core](/troubleshooting/opencode-connection/).
