# CodeCaptain-core Server

CodeCaptain se ejecuta sobre un servidor de CodeCaptain-core. De forma predeterminada inicia uno por ti, así que no tienes que hacer nada. Solo necesitas esta página si quieres apuntar CodeCaptain a un servidor que ya ejecutas, o gestionar el que él inicia.

## Cómo encuentra CodeCaptain un servidor

Cuando CodeCaptain arranca, busca un servidor en este orden:

1. reutilizar un servidor que ya inició
2. conectarse a uno externo si se lo indicaste (ver más abajo)
3. detectar automáticamente un servidor en el puerto predeterminado (`4096`)
4. de lo contrario, iniciar y gestionar el suyo propio

Si no hay nada configurado, el paso 4 ocurre automáticamente y ya tienes todo en marcha.

## Conéctate a un servidor que ya ejecutas

Define esto antes de iniciar CodeCaptain:

```bash
OPENCODE_HOST=http://localhost:4096 OPENCODE_SKIP_START=true codecaptain
```

- `OPENCODE_HOST` — la dirección completa de tu servidor de CodeCaptain-core, incluido el puerto (un valor como `http://localhost:4096`). No debe tener una ruta al final.
- `OPENCODE_SKIP_START=true` — le indica a CodeCaptain que no inicie su propio servidor.

Si solo necesitas cambiar el puerto, define `OPENCODE_PORT` en lugar de `OPENCODE_HOST`.

Si a `OPENCODE_HOST` le falta el puerto o tiene una ruta, CodeCaptain lo ignora y recurre a iniciar su propio servidor. Observa los registros de arranque por si aparece una advertencia `[config]` cuando una conexión que esperabas no ocurrió.

## Gestiona el servidor desde la CLI

```bash
codecaptain status
codecaptain logs
codecaptain restart
codecaptain stop
```

`codecaptain` por sí solo inicia el servidor en segundo plano. Añade `--foreground` para mantenerlo asociado a tu terminal.

## Iniciar CodeCaptain al iniciar sesión

Usa `startup enable` para instalar un servicio nativo de usuario. CodeCaptain usa `launchd` en macOS, `systemd --user` en Linux y Task Scheduler en Windows.

```bash
codecaptain startup enable
codecaptain startup status
codecaptain startup disable
```

Para proteger la UI, define la contraseña al habilitar el servicio:

```bash
CODECAPTAIN_UI_PASSWORD='secret' codecaptain startup enable
```

Para un servidor headless que arranca al iniciar sesión y se usa desde apps de escritorio o móviles, añade `--api-only` y un host alcanzable:

```bash
codecaptain startup enable --port 3000 --api-only --host 0.0.0.0 --ui-password secret
```

`startup enable` guarda una captura del entorno actual en el servicio para que el arranque se parezca más a ejecutar `codecaptain` desde la misma shell. Así conserva tokens de proveedores, `PATH`, configuración del agente SSH y otras variables CLI de auth/config. Usa `--no-env-snapshot` si quieres un entorno de servicio mínimo.

El servicio de inicio recuerda `--port`, `--host`, `--ui-password` y `--api-only`. El reinicio por CLI y el reinicio durante una actualización reutilizan esos ajustes guardados.

Para crear un enlace de conexión para otra app de CodeCaptain, usa:

```bash
codecaptain connect-url --port 3000 --server http://your-host:3000 --qr
```

Ejecuta `codecaptain connect-url --help` para ver todas las opciones del enlace, incluidas `--name`, `--lan`, `--server`, `--api-only`, `--ui-password` y `--qr`.

Puedes gestionar túneles de forma independiente para ese servicio en ejecución:

```bash
codecaptain tunnel start --port 3000
codecaptain tunnel stop --port 3000
```

Detener el túnel no reinicia el servicio ni la app.

## "CodeCaptain-core is restarting"

Mientras el servidor se inicia o reinicia, CodeCaptain muestra un estado "CodeCaptain-core is restarting" y pausa las solicitudes hasta que esté listo. Esto es normal justo después de iniciar o reiniciar. Si nunca desaparece, consulta [Conexión de CodeCaptain-core](/es/troubleshooting/opencode-connection/).
