# CodeCaptain-core Server

CodeCaptain працює поверх сервера CodeCaptain-core. За замовчуванням він сам запускає такий сервер, тож вам нічого робити не доводиться. Ця сторінка потрібна лише тоді, коли ви хочете спрямувати CodeCaptain на сервер, який ви вже запускаєте, або керувати тим, що він запускає.

## Як CodeCaptain знаходить сервер

Коли CodeCaptain стартує, він шукає сервер у такому порядку:

1. повторно використовує сервер, який уже запустив
2. підключається до зовнішнього, якщо ви це налаштували (див. нижче)
3. автоматично виявляє сервер на стандартному порту (`4096`)
4. інакше — запускає й керує власним

Якщо нічого не налаштовано, крок 4 відбувається автоматично, і все працює.

## Підключення до сервера, який ви вже запускаєте

Задайте це перед запуском CodeCaptain:

```bash
OPENCODE_HOST=http://localhost:4096 OPENCODE_SKIP_START=true codecaptain
```

- `OPENCODE_HOST` — повна адреса вашого сервера CodeCaptain-core, разом із портом (значення на кшталт `http://localhost:4096`). У кінці не повинно бути шляху.
- `OPENCODE_SKIP_START=true` — каже CodeCaptain не запускати власний сервер.

Якщо вам потрібно лише змінити порт, задайте `OPENCODE_PORT` замість `OPENCODE_HOST`.

Якщо в `OPENCODE_HOST` бракує порту або є шлях, CodeCaptain ігнорує його й повертається до запуску власного сервера. Слідкуйте за журналами запуску на предмет попередження `[config]`, якщо очікуване підключення не відбулося.

## Керування сервером із CLI

```bash
codecaptain status
codecaptain logs
codecaptain restart
codecaptain stop
```

`codecaptain` сам по собі запускає сервер у фоні. Додайте `--foreground`, щоб лишити його прив'язаним до вашого терміналу.

## Запуск CodeCaptain при вході в систему

Скористайтеся `startup enable`, щоб встановити нативний користувацький сервіс. CodeCaptain використовує `launchd` на macOS, `systemd --user` на Linux і Task Scheduler на Windows.

```bash
codecaptain startup enable
codecaptain startup status
codecaptain startup disable
```

Щоб захистити UI, задайте пароль під час увімкнення сервісу:

```bash
CODECAPTAIN_UI_PASSWORD='secret' codecaptain startup enable
```

Для headless-сервера, який стартує під час входу та використовується desktop або mobile застосунками, додайте `--api-only` і доступний host:

```bash
codecaptain startup enable --port 3000 --api-only --host 0.0.0.0 --ui-password secret
```

`startup enable` зберігає знімок поточного середовища в сервісі, щоб запуск був ближчим до ручного запуску `codecaptain` з тієї самої shell-сесії. Так зберігаються токени провайдерів, `PATH`, налаштування SSH agent та інші CLI-змінні для auth/config. Використайте `--no-env-snapshot`, якщо потрібне мінімальне середовище сервісу.

Startup-сервіс пам'ятає `--port`, `--host`, `--ui-password` і `--api-only`. CLI restart і restart під час оновлення повторно використовують ці збережені налаштування.

Щоб створити link підключення для іншого застосунку CodeCaptain, використайте:

```bash
codecaptain connect-url --port 3000 --server http://your-host:3000 --qr
```

Запустіть `codecaptain connect-url --help`, щоб побачити всі опції link, зокрема `--name`, `--lan`, `--server`, `--api-only`, `--ui-password` і `--qr`.

Тунелями для такого запущеного сервісу можна керувати окремо:

```bash
codecaptain tunnel start --port 3000
codecaptain tunnel stop --port 3000
```

Зупинка тунелю не перезапускає сервіс або застосунок.

## «CodeCaptain-core перезапускається»

Поки сервер запускається або перезапускається, CodeCaptain показує стан «CodeCaptain-core перезапускається» і призупиняє запити, доки той не буде готовий. Це нормально одразу після запуску чи перезапуску. Якщо стан так і не зникає, див. [Підключення до CodeCaptain-core](/uk/troubleshooting/opencode-connection/).
